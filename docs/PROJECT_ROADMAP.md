# 猫格 MIBT 项目现状与后续推进路线

> 文档用途：统一产品、开发、运营和商务合作的推进方向。
> 状态口径：文档明确区分“已经支持”“接入后生效”和“计划能力”，对外沟通时不得把计划能力描述为现有能力。
> 最近更新：2026-07-18

## 1. 项目概述

猫格 MIBT 是一个根据猫咪日常行为生成四维性格倾向和 16 种猫格类型的趣味测试产品。

产品当前的核心价值不是医学诊断，而是：

1. 帮助主人更有结构地观察猫咪行为。
2. 用容易理解的类型和内容改善人与猫的相处。
3. 生成适合社交传播的个性化结果。
4. 在积累真实产品数据后，为内容合作、用品推荐和商家联名提供入口。

产品长期可以形成以下闭环：

```text
用户进入测试
  → 建立猫咪档案
  → 完成行为观察
  → 获得猫格和相处建议
  → 分享专属结果
  → 朋友查看并开始测试
  → 积累反馈与匿名汇总数据
  → 优化题目、内容和合作场景
```

## 2. 当前产品定位

### 2.1 当前定位

- 面向普通养猫用户的趣味行为观察工具。
- 以移动端使用和社交分享为主要场景。
- 以“帮助理解相处偏好”为核心表达。
- 不宣称医疗、心理或动物行为诊断能力。
- 暂不以广告、支付或会员作为产品核心。

### 2.2 当前目标用户

- 第一次养猫、希望更了解猫咪的新手主人。
- 喜欢记录和分享猫咪内容的年轻养宠人群。
- 多猫家庭，希望理解不同猫咪相处节奏的用户。
- 猫咖、宠物店、领养机构等未来可能的线下合作方。
- 宠物玩具、用品和环境丰富化相关品牌。

### 2.3 产品表达边界

可以表达：

- “根据日常行为观察生成趣味性格倾向。”
- “结果帮助主人理解社交、探索、情绪回应和生活节奏。”
- “不同阶段和环境可能影响猫咪表现。”

不应表达：

- “精准诊断猫咪性格或疾病。”
- “测试结果可以替代兽医或行为咨询。”
- “某种类型一定适合或不适合某种食品、药物或治疗方式。”
- “某种类型比其他类型更聪明、更健康或更优越。”

## 3. 当前已经支持的功能

### 3.1 测试与档案

- 支持 16 题快速观察版。
- 支持 32 题完整观察版。
- 支持填写猫咪名字、年龄阶段和品种。
- 支持上传猫咪头像。
- 头像会在浏览器内压缩后保存。
- 档案默认只保存在当前设备浏览器中。
- 支持五档倾向选择：非常像左边、比较像左边、差不多、比较像右边、非常像右边。
- 支持返回上一题修改答案。
- 支持答题进度展示。

### 3.2 断点续答与历史结果

- 测试过程中自动保存当前题号和答案。
- 关闭页面后再次进入可以继续未完成测试。
- 用户可以主动删除未完成草稿。
- 最近完成的结果保存在当前浏览器。
- 支持从首页重新打开历史结果。
- 历史记录包含当次档案、答案、测试版本和校准题信息。
- 当前最多保留最近 6 条本地历史记录。

注意：当前没有账号系统，换设备、清理浏览器数据或使用无痕模式后，本地档案、草稿和历史结果可能丢失。

### 3.3 计分与校准

- 支持 E/I、S/N、T/F、J/P 四个维度计分。
- 支持反向题按实际字母语义计分。
- 答案强度会影响最终比例。
- 支持显示接近均衡、轻微倾向和明显倾向。
- 快速版出现维度平分时，会自动增加对应维度的校准题。
- 对计分逻辑和分享链接逻辑已有自动化测试。

当前限制：用户在校准题中仍选择完全中立时，该维度会继续显示接近均衡；四字母代码仍需要选择一侧作为结果展示，不能把代码本身理解为绝对结论。

### 3.4 结果内容

- 支持完整的 16 种猫格类型。
- 每种类型包含：
  - 中文类型名称。
  - 一句话性格描述。
  - 完整性格摘要。
  - 典型日常行为。
  - 性格优势。
  - 需要留意的表现。
  - 主人与猫的相处建议。
- 支持四维比例和倾向强度展示。
- 支持结果准确度反馈：很像、部分像、不太像。
- 支持根据性格生成三类生活用品方向和选购关键词。
- 当前用品建议明确标注为非医疗建议。
- 当前没有接入真实付费推广商品。

### 3.5 分享与传播

- 支持生成高清 PNG 结果海报。
- 海报包含猫咪头像、类型、四维比例和二维码。
- 支持下载结果海报。
- 支持调用浏览器或手机系统分享能力。
- 不支持系统分享时，会降级为复制分享文案和链接。
- 每次结果会生成可长期访问的结果链接。
- 分享结果链接只携带：
  - 猫咪名字。
  - 四字母类型。
  - 四维分数。
  - 测试版本。
- 分享链接不携带猫咪头像、主人资料和完整答案。
- 朋友打开分享链接后可以直接查看对应结果，并进入自己的测试。

隐私提醒：猫咪名字会出现在分享链接参数中。后续可以增加“分享时使用昵称/匿名”的开关。

### 3.6 类型内容页面

- 支持 16 种类型的独立静态页面。
- 支持类型索引页面。
- 每个类型页包含独立标题、描述、行为画像、优势、注意事项和相处建议。
- 支持在不同类型之间浏览。
- 支持从类型页进入测试。
- 页面使用静态生成，适合被搜索引擎收录。

### 3.7 数据与事件

当前已定义并记录的主要事件包括：

- 页面访问。
- 档案完成。
- 测试开始。
- 测试恢复。
- 答题进度节点。
- 添加校准题。
- 测试完成。
- 打开历史结果。
- 下载海报。
- 分享结果。
- 打开分享结果页。
- 点击用品建议。
- 提交结果准确度反馈。

默认情况下，事件只保存在当前浏览器，并推送到 `window.dataLayer`。

项目已预留 Umami 兼容的线上统计接入方式。填写以下环境变量后，事件才会进入真实线上统计系统：

```env
NEXT_PUBLIC_ANALYTICS_SCRIPT_URL=
NEXT_PUBLIC_ANALYTICS_WEBSITE_ID=
```

未填写时，不能宣称已经具备线上用户数据看板。

### 3.8 技术与部署

- Next.js 16。
- React 19。
- TypeScript。
- Tailwind CSS 依赖已存在，主要页面样式当前使用全局 CSS。
- 使用 `qrcode` 在浏览器生成结果二维码。
- 支持 Next.js 静态导出。
- 构建产物输出到 `out/`。
- 当前通过 GitHub `main` 分支部署到阿里云 ESA Pages。
- 当前不依赖服务端 API、SSR、ISR、数据库或对象存储。
- 已支持代码检查、计分测试、分享链接测试和生产构建。

常用命令：

```bash
pnpm dev
pnpm lint
pnpm test
pnpm build
```

## 4. 当前尚未支持的能力

以下能力不能作为现有产品对外承诺：

- 用户账号和登录。
- 跨设备同步。
- 云端猫咪档案。
- 商家后台。
- 内容管理后台。
- 真实商品库和库存同步。
- 淘宝、京东或其他联盟佣金链接。
- 支付、订单、会员、订阅和退款。
- 优惠券核销。
- 商家客户线索收集。
- 品牌联名主题配置后台。
- A/B 测试平台。
- 正式的线上数据看板，除非已经配置统计服务。
- 兽医或动物行为学诊断。
- AI 在线问诊或医疗建议。
- 微信小程序、App、PWA 离线安装。
- 多语言版本。

## 5. 后续推进原则

1. 先验证产品完成率、分享率和结果认可度，再进行商业化开发。
2. 先使用少量人工配置完成商家试点，再开发通用商家后台。
3. 商业内容必须明确标注，不把赞助内容伪装成测试结论。
4. 不出售用户原始档案、头像、完整答案或可识别个人的信息。
5. 医疗、药品、疾病和治疗相关内容必须保持严格边界。
6. 每次新增功能都应能够通过事件数据判断是否有效。
7. 保持快速版简单，不让商业内容干扰答题完成。

## 6. 阶段路线图

### 阶段 A：公开测试准备

目标：让产品可以稳定地面向真实用户开放，并获得可信的基础数据。

建议完成：

- 配置正式域名和 HTTPS。
- 配置真实线上统计服务。
- 增加隐私说明、使用条款、免责声明、关于我们和反馈入口。
- 增加一键删除全部本地数据。
- 增加分享时匿名或修改猫咪显示名称的选项。
- 增加 sitemap、robots.txt 和规范链接。
- 压缩社交预览图和页面静态资源。
- 验证微信内置浏览器、iPhone Safari 和 Android Chrome。
- 验证海报下载、二维码扫码和系统分享降级逻辑。
- 增加错误监控。
- 邀请少量真实养猫用户完成体验测试。

建议验收指标：

- 页面没有阻断测试的严重错误。
- 快速版完成率达到内部设定目标。
- 分享结果链接可以稳定打开。
- 海报二维码扫码成功率稳定。
- 结果反馈能够进入线上统计系统。

### 阶段 B：内容与自然增长验证

目标：证明产品可以通过分享和内容持续获得新用户。

建议完成：

- 为 16 种类型制作统一的小红书、朋友圈和短视频内容模板。
- 制作“社牛猫/社恐猫”“规律猫/随性猫”等维度内容。
- 为每个类型页补充常见问题。
- 根据搜索词制作行为观察文章。
- 为分享链接保留渠道参数。
- 建立“分享用户带来新测试”的归因统计。
- 优化朋友查看结果后的“我也测测”入口。
- 根据题目流失率调整题目长度和措辞。
- 根据“很像/不太像”反馈检查结果内容。

建议重点指标：

- 首页到开始测试转化率。
- 开始测试到完成测试转化率。
- 每道题的流失率。
- 快速版和完整版完成率差异。
- 结果分享率。
- 分享结果打开率。
- 分享带来的新测试数量。
- 结果认可度分布。
- 16 种结果分布是否异常集中。

### 阶段 C：商家合作准备

目标：不急于开发复杂平台，先准备可以人工交付的商家合作产品。

需要准备的商务材料：

- 一页项目介绍。
- 可直接体验的测试链接。
- 16 种结果示例。
- 分享海报示例。
- 用户流程图。
- 可提供的合作形式。
- 商家需要提交的素材清单。
- 数据报告样例。
- 试点周期和交付清单。
- 商业内容标注规范。
- 报价单、合同模板和验收标准。

需要准备的产品能力：

- 活动编号和商家编号。
- 商家专属入口链接。
- 商家 Logo、名称、主题色和活动文案配置。
- 商家专属分享海报。
- 商品或优惠权益配置。
- 渠道参数和点击事件。
- 活动开始与结束时间。
- 商业合作标识。
- 商家活动数据导出。
- 活动结束后的下线或替换策略。

第一批试点建议仍使用代码或配置文件人工管理，不必立即开发完整后台。

### 阶段 D：商家试点

目标：用 1～3 个真实合作验证商家是否愿意为测试带来的互动、内容或转化付费。

优先合作对象：

1. 宠物玩具和环境丰富化用品品牌。
2. 猫窝、猫爬架、取食玩具、饮水和清洁用品品牌。
3. 猫咖、宠物店和宠物展会。
4. 领养机构或公益活动。
5. 宠物内容账号和线下社群。

谨慎合作对象：

- 宠物药品。
- 疾病检测。
- 处方粮。
- 医疗和治疗服务。

涉及健康和医疗时，不能根据趣味性格类型直接给出治疗或购买结论，需要单独进行专业审核。

### 阶段 E：商业产品化

目标：在试点被验证后，把重复的人工交付能力产品化。

可能建设：

- 商家活动配置后台。
- 商品库和上下架管理。
- 优惠券发放与核销。
- 商家数据报表。
- 多品牌主题模板。
- 活动复制和版本管理。
- 商家成员权限。
- 合同周期和活动到期提醒。
- 白标版本。
- API 或嵌入式组件。
- 用户账号和跨设备档案，但需要重新设计隐私和数据架构。

## 7. 商家合作模式

### 7.1 品牌联名测试

形式：

- 测试首页展示“猫格 MIBT × 品牌名”。
- 使用有限度的品牌主题色和素材。
- 结果海报带联名标识。
- 测试核心题目和计分保持独立，不因赞助方修改结果。

适合：品牌活动、展会、周年活动、新品内容传播。

交付物：

- 联名入口页。
- 联名结果海报。
- 活动渠道链接。
- 基础活动数据报告。

### 7.2 结果页用品推荐

形式：

- 根据社交、探索和生活节奏推荐用品类型。
- 商家商品必须与对应用品方向一致。
- 商品卡片展示推荐依据。
- 明确标注推广、赞助或合作关系。

原则：

- 不因商家付费修改猫格类型。
- 不把普通商品描述为治疗方案。
- 不展示与结果无关的强制弹窗。
- 用户应能够关闭或跳过商业内容。

### 7.3 商家活动引流

形式：

- 为猫咖、宠物店或展会生成专属二维码。
- 用户测试后获得活动说明或优惠权益。
- 统计各入口的开始、完成、分享和权益点击。

适合：线下活动、门店拉新、社群互动、展位小游戏。

### 7.4 内容联合发布

形式：

- 与宠物内容账号联合制作 16 种猫格内容。
- 商家提供产品素材，猫格提供行为场景和类型内容。
- 内容统一导向测试或类型详情页。

适合：低开发成本的第一轮合作验证。

### 7.5 白标或授权版本

形式：

- 使用商家品牌名称、域名和视觉主题。
- 根据合同决定是否保留“猫格 MIBT”署名。
- 商家拥有独立活动数据和配置。

前提：必须先通过少量联名试点证明需求，避免过早建设复杂平台。

### 7.6 匿名行业洞察

形式：

- 仅提供无法识别个人或单只猫咪的汇总信息。
- 例如不同入口的完成率、类型占比和用品点击偏好。

禁止：

- 向商家出售头像、完整答案、猫咪档案或可识别用户的数据。
- 未经明确授权把数据用于新的目的。
- 用小样本结果制造“行业结论”。

## 8. 商家标准对接流程

### 8.1 商务确认

需要确认：

- 商家主体和联系人。
- 合作目标：曝光、互动、内容、门店引流或商品点击。
- 活动时间。
- 目标渠道。
- 预计访问量。
- 商业标识要求。
- 是否需要用户留资。
- 是否涉及优惠券、抽奖、支付、食品、药品或医疗内容。

涉及留资、抽奖、支付、医疗或其他高风险能力时，应单独进行合规和技术评估，不能直接沿用普通联名流程。

### 8.2 方案与范围

输出一份书面范围说明：

- 页面范围。
- 品牌露出位置。
- 可修改内容和不可修改内容。
- 商家素材清单。
- 数据指标。
- 开发与验收时间。
- 上线和下线方式。
- 修改次数。
- 数据保存和交付方式。

### 8.3 商家素材

商家通常需要提供：

- 品牌中英文名称。
- Logo 源文件和使用规范。
- 品牌色。
- 活动标题和说明。
- 商品图片、名称、卖点和目标链接。
- 推广或赞助标识文案。
- 优惠权益规则。
- 活动联系人。
- 素材版权和使用授权确认。

### 8.4 配置与开发

建议的数据结构至少包含：

```ts
type MerchantCampaign = {
  merchantId: string;
  campaignId: string;
  brandName: string;
  logoUrl?: string;
  themeColor?: string;
  title: string;
  description?: string;
  startsAt: string;
  endsAt: string;
  disclosure: string;
  channel?: string;
  products?: Array<{
    id: string;
    title: string;
    imageUrl?: string;
    targetUrl: string;
    recommendationBasis: string;
    typeCodes?: string[];
  }>;
};
```

初期可以使用 TypeScript 或 JSON 配置；合作数量增加后，再迁移到后台和数据库。

### 8.5 上线前验收

- 品牌名称和 Logo 正确。
- 商业合作标识清楚。
- 商品链接和渠道参数正确。
- 活动时间正确。
- 所有二维码可以扫码。
- 微信、iPhone 和 Android 能完成测试。
- 分享海报生成正常。
- 数据事件可以进入统计系统。
- 活动结束后的页面状态已确认。
- 医疗和健康相关表述已经过专业审核。

### 8.6 活动数据报告

建议向商家提供：

- 活动访问人数。
- 开始测试人数。
- 完成测试人数。
- 测试完成率。
- 结果分享次数和分享率。
- 分享结果打开次数。
- 商品或权益点击次数和点击率。
- 渠道来源分布。
- 设备和浏览器分布。
- 类型分布，仅在样本量足够且不涉及识别时提供。

计算口径应提前确定：

```text
测试开始率 = 开始测试人数 / 活动访问人数
测试完成率 = 完成测试人数 / 开始测试人数
结果分享率 = 分享结果人数 / 完成测试人数
推荐点击率 = 商品点击人数 / 查看结果人数
分享转化率 = 由分享链接开始测试人数 / 分享结果打开人数
```

报告必须说明统计时间、去重规则、异常流量处理和数据限制。

## 10. 产品数据与实验计划

### 10.1 当前应优先观察

- 快速版和完整版完成率。
- 每道题退出率。
- 校准题出现比例。
- 16 种结果分布。
- “很像/部分像/不太像”比例。
- 分享率和分享结果打开率。
- 分享页到新测试的转化率。
- 历史结果重新打开率。

### 10.2 题目优化

当样本量足够后，应检查：

- 是否存在几乎所有人都选同一侧的题目。
- 是否存在用户经常选择中立的题目。
- 是否存在导致大量退出的题目。
- 快速版和完整版结果是否一致。
- 同一只猫间隔一段时间复测是否基本稳定。
- 题目是否受到年龄、健康或环境的强烈干扰。

题目和计分修改必须记录版本。不同版本的数据不能直接混合比较。

### 10.3 建议的事件公共字段

后续事件建议统一附带：

- `event_version`
- `quiz_version`
- `mode`
- `type_code`
- `campaign_id`
- `merchant_id`
- `channel`
- `referrer_type`
- `is_shared_visit`
- `timestamp`

不要把头像、完整答案、联系方式直接写入普通分析事件。

## 12. 隐私、合规和信任

### 12.1 当前数据原则

- 档案、头像、答案、草稿和历史默认保存在当前设备。
- 分享链接不包含头像和完整答案。
- 线上统计未配置时，事件不会进入远端统计平台。
- 用户应能清空草稿、历史记录和全部本地数据。

### 12.2 商业合作原则

- 明确标注推广、赞助、联名或合作。
- 不因商家付费改变测试结果。
- 不暗示普通用品具有医疗效果。
- 不向商家提供用户原始档案。
- 不使用商家无权使用的图片、Logo 或内容。
- 活动规则、优惠条件和有效期必须清楚。

### 12.3 上线前需要补充的页面

- 隐私说明。
- 使用条款。
- 测试免责声明。
- 商业内容说明。
- 联系方式。
- 数据删除说明。

具体法律、广告、促销、抽奖、支付、食品、药品和医疗要求，应根据实际业务和上线地区请专业人士审核。

## 15. 风险与应对

### 风险一：测试很有趣，但没人分享

应对：优化结果的个性化程度、海报内容和分享后的朋友查看体验，而不是先增加更多题目。

### 风险二：结果认可度低

应对：分析“不太像”反馈、题目中立率、维度分布和快速/完整版本一致率；邀请专业人士审阅题目。

### 风险三：商家要求影响测试结论

应对：在合同和方案中明确题目、计分和结果独立，商家只影响品牌包装和合规的推荐区域。

### 风险四：过早开发复杂后台

应对：前 1～3 个商家采用人工配置交付，确认需求重复出现后再建设后台。

### 风险五：数据和隐私问题

应对：保持数据最小化、明确用途、提供删除方式，默认不上传头像和完整答案。

### 风险六：合作商品和性格不匹配

应对：每个商品必须配置推荐依据和适用类型，保留人工审核，不允许只按出价决定展示。

## 16. 建议的 90 天推进计划

### 第 1～2 周：公开测试基础

- 配置线上统计。
- 补隐私、条款、免责声明和数据清除。
- 完成手机和微信浏览器测试。
- 优化图片和加载性能。

### 第 3～4 周：小范围用户测试

- 邀请 30～100 位真实用户。
- 收集完成率、分享率和准确度反馈。
- 修正最明显的题目和内容问题。
- 确认快速版与完整版差异。

### 第 5～8 周：内容增长

- 发布 16 种类型内容。
- 建立小红书、朋友圈或短视频渠道参数。
- 测试不同分享文案和结果卡。
- 分析分享带来的新测试。

### 第 9～10 周：商家材料

- 制作一页项目介绍。
- 制作联名测试演示。
- 准备数据报告样例。
- 准备商家素材清单、范围说明和验收清单。

### 第 11～12 周：首个商家试点

- 选择与测试场景高度匹配的用品品牌、门店或猫咖。
- 使用人工配置完成一次小规模活动。
- 记录开发成本、沟通成本、用户体验和商家反馈。
- 根据试点决定是否开发商家后台。

## 17. 阶段完成定义

### 17.1 公开测试就绪

- 正式域名可访问。
- 隐私和免责声明完整。
- 核心流程在主要手机浏览器可用。
- 线上统计和错误监控生效。
- 测试、分享和历史记录没有阻断问题。

### 17.2 商家试点就绪

- 可以生成商家专属入口和渠道参数。
- 可以配置品牌素材和商业标识。
- 可以配置相关商品或活动权益。
- 可以提供一份口径清晰的数据报告。
- 有书面合作范围和验收清单。

### 17.3 商业产品化就绪

- 已完成至少 1～3 个真实试点。
- 已确认商家愿意为哪一种价值付费。
- 重复需求明确。
- 后台、数据库或支付的投入有真实需求支撑。
- 隐私、合同和运营流程能够承担规模化合作。

## 18. 下一步建议

当前最优先的工作不是广告或支付，而是：

1. 配置真实线上统计服务。
2. 补齐隐私说明、免责声明和本地数据清除入口。
3. 完成微信、iPhone 和 Android 真机测试。
4. 增加 sitemap、robots.txt 和正式域名配置。
5. 邀请第一批真实用户并分析结果反馈。
6. 建立测试版本号和数据口径。
7. 制作第一版项目介绍和商家合作演示。
8. 选择一个低风险、强相关的商家做人工配置试点。

在完成真实用户验证之前，不建议优先开发支付、会员、复杂商家后台或大规模广告系统。

## 19. 文档维护规则

- 每个阶段完成后更新“当前已经支持的功能”。
- 新增商业能力时同步更新合作边界和验收清单。
- 题目或计分变化时记录版本和生效日期。
- 统计事件变化时更新事件清单和计算口径。
- 对外材料只引用已经上线并经过验证的能力。
